# Codebase Concerns

**Analysis Date:** 2026-01-30

## Tech Debt

**Legacy File-Based Indices Migration Incomplete:**
- Issue: Multiple deprecated functions remain for backwards compatibility during migration from file-based to PostgreSQL storage
- Files: `fitz_ai/core/paths/indices.py`
- Impact: Code paths for old vocabulary (YAML), sparse_index, and entity_graph (SQLite) are dormant but still present; adds cognitive overhead and potential maintenance burden
- Fix approach: Audit migration completion status, remove deprecated compatibility functions once migration is verified complete across all deployments (1-2 phases)

**VDB Delete Operation Placeholder:**
- Issue: Table deletion not properly implemented - currently a no-op, rows overwrite on re-ingest
- Files: `fitz_ai/structured/schema.py` (line 248), `fitz_ai/structured/ingestion.py` (line 289)
- Impact: No way to remove old rows from structured data without manual deletion; can cause stale data in long-running systems
- Fix approach: Implement proper filtered delete when vector DB supports it, or batch delete with collection recreation

**Content Type Detection Assumption:**
- Issue: Enrichment pipeline assumes all projects are code projects for artifact generation
- Files: `fitz_ai/ingestion/enrichment/pipeline.py` (line 233)
- Impact: Non-code projects (documentation, knowledge bases) may not get appropriate enrichments; reduces quality of hierarchy summaries for these domains
- Fix approach: Detect actual content types in project during ingestion startup, pass to enrichment pipeline

**Async Cloud Client Not Implemented:**
- Issue: Async integrations (LangChain, LlamaIndex) delegate to sync cloud client
- Files: `fitz_ai/integrations/langchain/runnable.py` (line 217), `fitz_ai/integrations/llamaindex/query_engine.py` (line 273)
- Impact: Async context cannot use cloud cache for answers; blocks I/O during cloud operations
- Fix approach: Create AsyncCloudClient wrapper, implement proper async HTTP in cloud module

## Known Bugs

**pgserver Windows Crash Recovery Hangs:**
- Symptoms: 30-second delay on Windows when pgserver crashes or is forcefully killed
- Files: `fitz_ai/storage/postgres.py` (covered by fork)
- Trigger: Ctrl+C during database operations, or pgserver crash with corrupted recovery
- Workaround: Use fitz-pgserver fork which moves logs to temp directory (already in dependencies)
- Status: FIXED in fitz-pgserver 0.1.5+, verified in pyproject.toml

**Stale Lock Files on Windows:**
- Symptoms: pgserver fails to start after unclean shutdown; "could not open file ./log" errors
- Files: `fitz_ai/storage/postgres.py` (lines 113-151, 153-196)
- Trigger: Process killed without graceful shutdown, lock files remain in pgdata
- Workaround: Implemented 10s timeout on stale lock cleanup, automatic nuclear recovery (delete pgdata)
- Status: MITIGATED with aggressive cleanup logic

**Incomplete Error Context in Cloud Cache:**
- Symptoms: Cloud cache failures (network, encryption, API) silently downgrade to non-cached retrieval
- Files: `fitz_ai/cloud/client.py` (error handling catches broad exceptions)
- Trigger: Network transient, org_key mismatch, API timeout
- Workaround: Check logs for cloud errors, may be performance-impacting silent fallback
- Status: ACCEPTABLE for production (graceful degradation), but limits observability

## Security Considerations

**Encryption Key Management:**
- Risk: org_key stored in memory during runtime; if process memory is swapped to disk, encryption key could leak
- Files: `fitz_ai/cloud/crypto.py` (line 58), `fitz_ai/cloud/client.py` (line 73)
- Current mitigation: org_key NEVER transmitted to server, server cannot decrypt cached data, encryption uses industry-standard AES-256-GCM with authenticated nonces
- Recommendations:
  - Document that users should lock memory in security-critical deployments (`mlock` on Linux)
  - Consider adding memory-locking option via `ctypes` for sensitive environments
  - Add warning in docs about VM memory dumps

**SQL Generation LLM Output Trust:**
- Risk: SQL queries generated by LLM are parsed but not fully validated; malformed WHERE clauses could bypass filtering or cause errors
- Files: `fitz_ai/structured/sql_generator.py` (lines 68-95, JSON parsing at line 125)
- Current mitigation: SQLQuery dataclass validates column names against schema, WHERE conditions parsed to dicts
- Recommendations:
  - Add whitelist validation for aggregation functions in select statements
  - Test generated SQL against schema validation before execution
  - Add length limits on generated queries (prevent resource exhaustion)
  - Log all generated SQL queries for audit trail

**API Key Exposure in Headers:**
- Risk: API keys passed in Bearer tokens are in HTTP headers; logs might capture full headers
- Files: `fitz_ai/core/http.py` (line 194), `fitz_ai/llm/providers/*.py` (header extraction)
- Current mitigation: Keys loaded from environment variables, not hardcoded
- Recommendations:
  - Ensure HTTP client logs don't include Authorization headers (verify httpx default behavior)
  - Add mask function for logs that sanitize Bearer tokens
  - Document that httpx.Client should not log response bodies when keys are involved

**Collection Name Sanitization:**
- Risk: Collection names converted to database names; insufficient validation could allow SQL injection or unintended database names
- Files: `fitz_ai/storage/postgres.py` (line 199-206, database creation at line 285)
- Current mitigation: Regex whitelist `[a-zA-Z0-9_]`, forced lowercase, prefix added if starts with number
- Recommendations:
  - Add test case for edge cases (very long names >63 chars, unicode characters)
  - Document that collection names with special characters will be transformed
  - Verify PostgreSQL identifier length limits are enforced

## Performance Bottlenecks

**Large File Processing Memory:**
- Problem: Docling parser loads entire files into memory; large PDFs (100MB+) cause memory spikes
- Files: `fitz_ai/ingestion/parser/plugins/docling.py` (document loading)
- Cause: DocumentConverter.convert_all() loads all elements before parsing
- Improvement path:
  - Stream parsing in chunks for multi-MB documents
  - Add memory limits and fallback to simpler parser for very large files
  - Document memory requirements in README

**Entity Graph Expansion in Retrieval:**
- Problem: Multi-entity queries iterate through entity_chunks table; O(n*m) complexity for k entities
- Files: `fitz_ai/engines/fitz_rag/retrieval/steps/strategies/semantic.py` (entity expansion)
- Cause: No precomputation of entity co-occurrence; sequential lookups
- Improvement path:
  - Cache entity relationship graph on engine startup
  - Use PostgreSQL native array operations for batch entity lookups
  - Profile with 10k+ entities to identify actual bottleneck

**Enrichment Pipeline Sequential Processing:**
- Problem: Chunks processed one batch at a time (15 per LLM call); large ingestions take hours
- Files: `fitz_ai/ingestion/enrichment/bus.py` (batching logic)
- Cause: One LLM call per batch, no parallel processing
- Improvement path:
  - Support configurable batch size (currently hardcoded 15)
  - Add async LLM support for concurrent batch enrichment
  - Profile ingestion of 10k+ documents to measure impact

**Vector DB Connection Pool Initialization:**
- Problem: Creating connection pool per collection on first query causes initial request latency
- Files: `fitz_ai/storage/postgres.py` (line 350-380)
- Cause: Lazy pool creation; collection metadata fetched on first use
- Improvement path:
  - Pre-create pools during engine initialization for known collections
  - Cache pool metadata in memory (expires on collection delete)

## Fragile Areas

**PostgreSQL Recovery Logic (Nuclear Option):**
- Files: `fitz_ai/storage/postgres.py` (lines 153-196)
- Why fragile: Cascading fallbacks (unlock → kill processes → delete pgdata → reinit); if any step fails, recovery aborts
- Safe modification:
  - Add logging at each recovery stage (already done)
  - Test recovery on corrupted pgdata manually before changes
  - Verify pgdata deletion succeeds before reinit attempt
- Test coverage: `tests/unit/test_postgres_recovery.py` provides scenarios (512 lines)

**Cloud Cache Encryption/Decryption Path:**
- Files: `fitz_ai/cloud/crypto.py` (lines 91-115)
- Why fragile: Broad exception catching (line 114) masks real decryption failures; wrong org_key produces unhelpful error
- Safe modification:
  - Distinguish between authentication (wrong key) vs. corruption (tampered blob)
  - Add specific exception types for different failure modes
  - Log org_key hash (not key itself) for debugging
- Test coverage: Unit tests verify encrypt/decrypt round-trip, but not key mismatch scenarios

**Schema Validation in Structured Data:**
- Files: `fitz_ai/structured/schema.py`, `fitz_ai/structured/executor.py`
- Why fragile: Schema inferred from data during first ingest; changes to table structure require manual re-registration
- Safe modification:
  - Add schema versioning and migration detection
  - Test with schema changes (adding columns, type changes)
  - Document schema evolution best practices
- Test coverage: `tests/unit/structured/test_schema.py` (436 lines) covers basic cases; missing evolution scenarios

**Ingestion State Manager Singleton:**
- Files: `fitz_ai/ingestion/state/manager.py`
- Why fragile: Global state for tracking ingestion progress; concurrent ingestions could interfere
- Safe modification:
  - Use collection-scoped state instead of global
  - Test with concurrent ingestions (missing in current suite)
  - Add lock around state mutations
- Test coverage: No concurrency tests

## Scaling Limits

**Vector DB Query Limit:**
- Current capacity: pgvector HNSW index performs best with <10M embeddings per collection
- Limit: Beyond 10M, query performance degrades logarithmically; index becomes memory-intensive
- Scaling path:
  - For 10M-100M: Shard across multiple PostgreSQL databases (manual)
  - For 100M+: Move to Pinecone/Qdrant serverless (configure in vector_db config)
  - Document sharding strategy in operational guide

**Enrichment LLM Cost:**
- Current capacity: Every chunk costs 1 LLM call (summarization) = $0.01/1000 chunks at Claude 3.5 Sonnet rates
- Limit: Large corpus (100k+ documents) ingestion costs $1000+ in API calls
- Scaling path:
  - Make enrichment module optional via config flag
  - Implement sampled enrichment (e.g., 1 in 100 chunks)
  - Cache embeddings to skip re-enrichment on incremental ingest

**Connection Pool Limits:**
- Current capacity: psycopg_pool default 10 connections per collection
- Limit: Concurrent users > 50 per collection will saturate pool
- Scaling path:
  - Configure pool size in storage config (not exposed yet)
  - Monitor connection pool metrics in production
  - Add queue timeout for pool exhaustion

## Scaling Limits (continued)

**API Rate Limiting:**
- Current: No built-in rate limiting on FastAPI endpoints
- Impact: Single malicious client can exhaust LLM quota or vector DB
- Fix: Add rate limiter middleware (per-IP, per-org), configure per-endpoint budgets

## Dependencies at Risk

**Docling Version Pin:**
- Risk: docling>=2.0 required; major version bumps may break parsing
- Impact: PDF parsing failures on docling 3.0 if API changes
- Migration plan: Pin to docling==2.x, test 3.0 in separate branch before upgrade

**psycopg Major Version:**
- Risk: psycopg 3.x changed async API significantly from 2.x
- Impact: If async cloud client implemented, must use psycopg[binary]>=3.1 consistently
- Migration plan: Documented in CLAUDE.md; all code uses psycopg 3.x already

**Cryptography Library:**
- Risk: cryptography>=41.0 required; 40.x has known vulnerabilities
- Impact: Encryption key derivation could be weakened in old versions
- Migration plan: Pin to latest, auto-update in CI

## Missing Critical Features

**No Query Result Caching at Application Level:**
- Problem: Same query asked twice hits LLM twice (cloud cache helps, but adds latency)
- Blocks: Power users with repeated queries, batch analysis workflows
- Solution: Add in-memory LRU cache of recent answers (5min TTL)

**No Rate Limiting on LLM Calls:**
- Problem: Runaway query loops can exhaust API quota without warning
- Blocks: Production deployments without cost controls
- Solution: Add per-query LLM token budget, configurable quota per engine

**No Structured Data Schema Validation on Insert:**
- Problem: Malformed rows inserted silently; corrupts search results
- Blocks: Direct CSV import workflows that need data quality gates
- Solution: Add JSONSchema validation before upsert in table store

**No Audit Trail for Cloud Cache Operations:**
- Problem: No log of what was cached, when, by whom
- Blocks: Compliance requirements (healthcare, finance)
- Solution: Add audit log to cloud cache client (separate from main logs)

## Additional Observations

**Cloud Crypto Nonce Generation:**
- Using `os.urandom(12)` for 96-bit nonces is correct for AES-GCM
- Prepending nonce to ciphertext follows standard practice
- No issues detected; implementation is sound

**HTTP Error Handling:**
- Comprehensive error mapping (401 → AuthenticationError, 429 → RateLimitError)
- Rate limit detection enables smart retry strategies
- No automatic retries implemented (intentional; caller decides)

**File Handle Management:**
- All file I/O uses context managers (`with` statements)
- No detected resource leaks
- Good practice throughout codebase

**Global Singletons:**
- PostgresConnectionManager is singleton (thread-safe with _lock)
- EngineRegistry is singleton (thread-safe)
- Signal handlers registered once (_signal_handlers_registered flag prevents duplicates)
- Architecture is sound for production use
