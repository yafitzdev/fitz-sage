## Architecture

### Explored Approaches
**1. Native Engine Integration**
- Description: The plugin lives directly inside `fitz_ai/engines/chat_openai/`. It implements the same abstract base class or protocol as `fitz_krag`. Configuration is loaded via `.fitz/config.yaml` referencing a new engine name.
- Pros: Reuses existing dependency injection and logging infrastructure, Integration with CLI (`fitz init`, `chat` commands) without custom CLI glue code
- Cons: Ignoring the import restrictions (e.g., accidentally importing from `rag/`) causing CI failure

**2. Adapter Layer / Wrapper Service**
- Description: You create a generic wrapper class that sits between the core engine interface and the OpenAI client, adding a layer of indirection to "standardize" future providers.
- Pros: Theoretically easier to swap backends later
- Cons: Introduces unnecessary complexity for a single provider, Makes debugging harder when `contract_map` fails due to adapter misconfiguration rather than logic errors, Requires maintaining the wrapper contract in addition to the engine implementation

**3. Standalone Microservice**
- Description: OpenAI logic runs as a separate Docker container communicating via HTTP/gRPC with the main `fitz-ai` process.
- Pros: Isolation of OpenAI library versioning from the core app
- Cons: Network latency, serialization overhead, Breaking the "hybrid Python engine" requirement (which implies local execution context), Requires managing a second service deployment lifecycle, separate secrets management, and network security policies

### Recommended: Native Engine Integration

This is the only approach that satisfies the constraint "Directory structure must follow existing engine patterns" and allows the mandatory `python -m tools.contract_map` validation to function correctly. Approach 2 creates artificial complexity for a single provider, violating the principle of least surprise in an established codebase. Approach 3 violates the core requirement of a "hybrid Python engine implementation" and ignores the CI workflow constraints (`ubuntu-latest`, Python 3.10+) which expect local execution.

*This task produces four core deliverables (provider, config, test, manifest update) — the challenge is strict adherence to existing protocol contracts and import restrictions, not building new infrastructure.*

## Design

### Architectural Decision Records
**ADR: ADR-001: Secret Resolution Priority**
- Decision: Runtime secrets resolved via Environment Variables first; secrets.yaml is only for local dev fallback and explicitly excluded from version control.
- Rationale: Aligns with CI workflow (ci.yml) and Secret Management patterns. Ensures CI/CD safety without untracked file dependencies in production.
- Alternatives: Using secrets.yaml in production — rejected because .gitignore excludes it and CI environment cannot access local untracked files.

**ADR: ADR-002: Config Schema Mirroring with Nulls**
- Decision: .fitz/config/openai_chat.yaml will mirror keys of .fitz/config/fitz_rag.yaml (e.g., chat, embedding, vector_db) but explicitly set unused fields to null.
- Rationale: Inferred from existing config structure. Guarantees compatibility with existing configuration loaders without modifying the core loader logic.
- Alternatives: Creating a disjointed schema specific to chat-only — rejected because risks deserialization errors in the core engine.

**ADR: ADR-003: Test Isolation via Mocking**
- Decision: tests/unit/test_chat_openai.py must use unittest.mock.patch to intercept openai.Client.create calls; no real network requests are allowed during unit tests.
- Rationale: Aligns with isolation expectations of the test infrastructure pattern. Ensures CI stability without external API dependencies.
- Alternatives: Using live API calls during unit tests — rejected because introduces flakiness and violates isolation expectations of fitz_ai.storage.postgres pattern.
